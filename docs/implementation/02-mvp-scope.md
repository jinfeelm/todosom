# MVP 범위와 로드맵

## MVP 정의

TodoSom MVP는 서버 없이 로컬 기기에서 핵심 사용자 루프를 검증하는 앱이다. 첫 목표는 기능을 많이 넣는 것이 아니라 사용자가 첫 투두 완료 후 펫 반응과 진화 기대감을 느끼는지 확인하는 것이다.

## 필수 기능

| 기능 | 설명 | 완료 기준 |
| --- | --- | --- |
| 투두 생성/완료 | 오늘 할 일을 빠르게 작성하고 체크한다 | 제목, 날짜, 카테고리, 완료 상태가 저장된다 |
| 사용자 카테고리 | 사용자가 직접 분류를 만든다 | 카테고리마다 7개 결 중 하나를 연결한다 |
| 7개 결 연결 | 몰입, 창작, 배움, 돌파, 돌봄, 연결, 정돈 | 완료 시 연결된 결 점수 이벤트가 생긴다 |
| 핵심 씨앗 | 하루 1개에서 3개의 중요한 할 일을 고른다 | 날짜별 3개 초과 선택이 막힌다 |
| 펫방 | 고해상도 도트 펫과 방을 보여준다 | 펫, 대사, 오늘의 결 요약이 보인다 |
| 말 걸기 | 상태 기반 짧은 대사를 표시한다 | 버튼 탭 시 상황에 맞는 대사가 바뀐다 |
| 성장기 진화 | 결 분포에 따라 기본 펫이 성장기 형태로 진화한다 | 조건 충족 시 진화 modal과 기록이 남는다 |
| 도감/기록의 방 | 이전 여정과 진화 기록을 보관한다 | 새 여정 후 이전 펫 카드가 남는다 |
| 상점 placeholder | 향후 BM 위치를 확보한다 | 결제 없이 비활성 안내만 보여준다 |

## 후순위 기능

| 기능 | 이유 | 준비만 할 것 |
| --- | --- | --- |
| 반복 투두 | MVP 핵심 루프 검증 후 필요 | `repeat_rule` 필드는 후속 migration으로 둔다 |
| 메모/서브태스크 | 입력이 무거워질 수 있음 | 도메인 모델에 즉시 넣지 않는다 |
| 방 꾸미기 풀기능 | 제작량이 과함 | room layer, inventory 구조만 설계한다 |
| Supabase 동기화 | 로컬 루프 검증 후 필요 | id, timestamps, user_id 구조는 확장 가능하게 둔다 |
| RevenueCat 결제 | 초기에는 BM 검증 전 | product_id, entitlement 개념만 문서화한다 |
| 푸시 알림 | 핵심 루프 후 리텐션 실험 | local notification 권한 UX는 별도 단계 |
| AI 자유 채팅 | 운영과 안전성 부담 큼 | 상태 기반 대사만 사용한다 |

## 제외 기능

- 소셜, 친구, 랭킹: 비교 스트레스와 어뷰징을 키울 수 있다.
- 유료 랜덤 알, 확률형 가챠: 초기 신뢰에 불리하다.
- 진화 속도 2배권: 자기관리 성과를 돈으로 사는 느낌을 준다.
- 리셋권 판매: 실패와 재시작 욕구를 과금화한다.
- 펫 사망: TodoSom의 복귀 정서와 충돌한다.
- 팀 협업 기능: 개인용 앱의 포지셔닝을 흐린다.
- 강한 연속 기록 패널티: 복귀를 어렵게 만든다.

## 0단계: 로컬 프로토타입

| 목표 | 포함 기능 | 완료 기준 |
| --- | --- | --- |
| 핵심 루프 검증 | 투두 생성, 완료, 결 점수, 펫 반응, 진화 | 서버 없이 기기에서 한 바퀴가 돈다 |
| 기본 UI 검증 | 오늘, 펫방, 기록, 상점 placeholder | 하단 탭 이동이 자연스럽다 |
| 더미 아트 적용 | 솜몽과 성장기 placeholder | 펫이 화면 중심에 보인다 |

### 0단계 수용 기준

- 앱 첫 실행 시 온보딩 화면이 뜬다.
- 첫 투두를 만들고 완료하면 솜알이 솜몽으로 깨어난다.
- 결 점수 이벤트가 DB에 저장된다.
- 펫방에서 오늘 쌓인 결과 펫 대사를 확인할 수 있다.
- 테스트 데이터로 성장기 진화 modal을 재현할 수 있다.

## 1단계: 알파 MVP

| 목표 | 포함 기능 | 완료 기준 |
| --- | --- | --- |
| 친한 사람 테스트 | 온보딩, 카테고리, 핵심 씨앗, 말 걸기, 도감 | 테스터가 첫 진화까지 간다 |
| 고퀄 아트 1차 | 솜몽 idle/talk/happy/eat/sleep | 펫방이 매력적으로 보인다 |
| 로컬 푸시 | 오늘의 씨앗 알림 | 알림 허용 흐름을 테스트한다 |

### 1단계 수용 기준

- 실제 사용자가 설명 없이 첫 할 일을 만들 수 있다.
- 핵심 씨앗 제한과 완료 보상이 이해된다.
- 첫 진화 결과가 사용자의 결 분포와 납득 가능하게 연결된다.
- 기록의 방에서 이전 여정의 의미가 사라짐이 아니라 보관으로 느껴진다.

## 2단계: 베타

| 목표 | 포함 기능 | 완료 기준 |
| --- | --- | --- |
| 계정/동기화 | Supabase Auth, Sync, RLS | 기기 변경에도 데이터 복구가 가능하다 |
| 분석 | 핵심 이벤트, D1/D7 지표 | 이탈 구간이 보인다 |
| 스토어 준비 | 아이콘, 스크린샷, 약관, 개인정보 | TestFlight/내부테스트가 가능하다 |

## 3단계: 출시

| 목표 | 포함 기능 | 완료 기준 |
| --- | --- | --- |
| 공개 출시 | 무료 기본 경험 완성 | 스토어 심사를 통과한다 |
| 초기 BM | 유료 유년기 펫 1종 또는 방 테마 1종 | 구매 흐름이 검증된다 |
| 운영 루프 | 피드백 수집, 핫픽스, 콘텐츠 업데이트 | 첫 달 리텐션 개선 사이클이 돈다 |

## MVP 우선순위 규칙

1. 핵심 루프를 강화하는가?
2. 사용자의 감정적 부담을 줄이는가?
3. 펫 애착을 키우는가?
4. 지금 팀이 감당할 수 있는 제작량인가?
5. 장기 BM 확장성을 데이터 구조로 열어두는가?

위 질문에서 1번 또는 2번에 답하지 못하는 기능은 MVP에서 제외한다.
