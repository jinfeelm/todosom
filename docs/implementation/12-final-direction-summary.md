# TodoSom 방향 최종 총정리

## 이 문서의 역할

이 문서는 지금까지 합의한 TodoSom의 방향을 한 장으로 묶은 최종 정리다. 백서의 큰그림, 구현 문서의 세부 기준, 1인 예비 창업자의 현실적인 실행 조건, 무료 베타부터 첫 BM까지의 목표를 서로 충돌하지 않게 연결한다.

## 핵심 결론

TodoSom은 단순히 귀여운 투두 앱이 아니다. 사용자의 작은 약속과 완료 경험을 `결`로 번역하고, 그 결이 솜뭉치 펫의 반응과 진화로 남는 앱이다.

초기 목표는 많은 기능을 만드는 것이 아니라, 사용자가 다음 감정을 느끼게 하는 것이다.

> 내가 한 일이 이 아이에게 남았다.

이 감정이 살아 있으면 앱은 계속 키울 가치가 있다. 이 감정이 약하면 기능을 많이 붙여도 제품은 흐려진다.

## 백서의 위치

`todosom_whitepaper_v0_1.docx`는 첫 버전에 전부 구현할 목록이 아니다. TodoSom이 커져서 수익화되고, 사업체가 생기고, 함께 일할 직원이 생겼을 때에도 제품의 방향을 잃지 않기 위한 큰그림 문서다.

백서가 고정하는 것은 다음이다.

- 핵심 루프: 할 일 -> 결 -> 펫 반응 -> 진화 -> 기록
- 실패한 사용자를 벌하지 않는 정서
- 무료 사용자도 핵심 성장과 기록의 재미를 충분히 경험한다는 원칙
- 유료 상품은 성능이 아니라 외형, 방, 서사, 연출 확장이라는 BM 윤리
- 모두 같은 기본 솜에서 시작한다는 브랜드 중심성

백서가 열어두는 것은 다음이다.

- 실제 출시 순서
- 펫 이름과 세부 대사
- 유료 상품의 가격과 출시 타이밍
- 서버, 계정, 결제, 운영 도구의 구체 구현 방식

## 1차 MVP의 정의

1차 MVP는 로컬 프로토타입만이 아니다. 무료 베타로 실제 사용자를 만나고, 첫 BM 상품 1개를 검증하는 순간까지를 포함한다.

1차 MVP 한 문장 정의:

> 무료로 충분히 사랑받을 수 있는 TodoSom 핵심 루프를 출시하고, 사용자가 돈을 내도 납득할 첫 감성 확장 상품 1개를 검증하는 것.

## 단계별 제품 경계

### Phase 0: 로컬 감정 루프 프로토타입

목표는 TodoSom의 감정 루프가 작동하는지 확인하는 것이다.

필수:

- 온보딩
- 첫 씨앗 생성과 완료
- 솜알에서 솜몽으로 부화
- 오늘 화면
- 펫방
- 상태 기반 대사
- SQLite local-first 저장
- 점수와 진화 순수 로직 테스트

제외:

- 계정
- 클라우드 동기화
- 실제 결제
- 소셜
- 방 꾸미기 풀기능
- AI 자유 채팅

### Phase 1: 무료 베타

목표는 실제 사용자가 다시 여는지 확인하는 것이다.

필수:

- TestFlight 또는 Google Play Internal Testing 배포
- 기본 분석 이벤트
- crash/error 추적
- 로컬 푸시 또는 오늘의 씨앗 알림
- 첫 성장기 진화
- 기록의 방
- 복귀 경험
- 개인정보 처리방침과 지원 채널

### Phase 2: 무료 공개 베타

목표는 작은 공개 사용자 풀에서 메시지, 안정성, 리텐션을 확인하는 것이다.

필수:

- 스토어 공개 또는 제한 공개
- 앱 아이콘, 스크린샷, 설명문
- 온보딩 퍼널 개선
- 주요 기기 QA
- 데이터 migration 안정화
- 사용자 피드백 수집 루프

### Phase 3: 첫 BM 검증

목표는 감성 확장에 돈을 낼 사람이 있는지 작게 확인하는 것이다.

첫 BM은 상품 1개만 검증한다. 추천 1순위는 `비 오는 공부방 테마`다. 대안은 `별솜 새 여정 팩`이다.

추천 1순위가 방 테마인 이유:

- 무료 핵심 진화 경험을 건드리지 않는다.
- 제작 범위가 비교적 작고 명확하다.
- 펫방 애착을 강화한다.
- 구매하지 않아도 앱이 부족해 보이지 않는다.
- TodoSom의 감성 가치와 잘 맞는다.

## 스택 판단

문서에 제안된 스택은 실제 배포급 퀄리티를 만들 수 있는 스택이다.

| 영역 | 선택 | 판단 |
| --- | --- | --- |
| 앱 | React Native + Expo | iOS/Android 동시 출시와 1인 개발에 적합 |
| 언어 | TypeScript | 도메인 규칙이 많은 앱이라 사실상 필수 |
| 라우팅 | Expo Router | 탭 기반 앱과 온보딩 흐름에 적합 |
| 로컬 DB | expo-sqlite | local-first MVP에 적합 |
| 상태 | Zustand | UI 상태와 view model 관리에 충분 |
| 빌드/배포 | EAS Build | TestFlight, Internal Testing, 스토어 배포 흐름에 적합 |
| 푸시 | Expo Notifications | 베타 단계 리텐션 실험에 적합 |
| 동기화 | Supabase Auth + RLS | 베타 이후 계정/복구 기능에 적합 |
| 결제 | RevenueCat | 첫 BM 검증과 구매 복원 처리에 적합 |
| 고급 2D 연출 | React Native Skia | 진화 연출과 펫방 이펙트에 적합하되 초반 남용 금지 |

중요한 판단:

- 스택은 바꾸지 않는다.
- 처음부터 Supabase, RevenueCat, Skia를 모두 붙이지 않는다.
- Phase 0은 Expo, TypeScript, Expo Router, expo-sqlite, Zustand만으로 간다.
- Supabase와 RevenueCat은 무료 핵심 루프가 검증된 뒤 붙인다.

## 1인 바이브코딩 실행 원칙

사용자는 비개발자, 비디자이너, 1인 예비 창업자라는 조건에서 시작한다. 이 조건은 약점이지만, 범위를 잘 자르면 실행 가능한 조건이기도 하다.

지켜야 할 원칙:

- 큰 앱을 만들지 말고 작은데 선명한 앱을 만든다.
- AI에게 "앱 전체 만들어줘"라고 시키지 않고 작은 작업 단위로 시킨다.
- 기능 추가보다 QA 반복을 우선한다.
- 펫 10마리보다 솜몽 1마리의 완성도를 우선한다.
- 서버, 결제, 소셜, AI 채팅은 검증 전까지 미룬다.
- 출시급 퀄리티는 코드보다 체크리스트와 반복 테스트에서 나온다고 본다.

좋은 AI 요청 예시:

- `scoreTodoCompletion`의 단위 테스트를 만들어줘.
- 핵심 씨앗이 하루 3개를 넘지 않도록 repository에서 막아줘.
- 온보딩에서 첫 씨앗 생성 후 완료하면 egg가 baby로 바뀌게 해줘.
- 펫방 말 걸기 버튼이 dialogue condition에 따라 문구를 바꾸게 해줘.

나쁜 AI 요청 예시:

- 귀여운 투두 앱 전체 만들어줘.
- 다마고치 같은 앱 만들어줘.
- 결제랑 서버랑 푸시랑 다 붙여줘.

## 배포급 품질 기준

TodoSom의 품질 병목은 스택이 아니라 다음에 있다.

- 아트 퀄리티
- 펫방의 감정적 매력
- 데이터 무결성
- migration 안정성
- 작은 화면에서의 UI 완성도
- 사용자를 탓하지 않는 문구
- 반복 가능한 QA
- 스토어 심사와 개인정보 대응

출시 후보는 최소한 다음을 만족해야 한다.

- 앱을 껐다 켜도 투두, 펫, 점수, 진화 기록이 유지된다.
- 완료 이벤트가 중복 생성되지 않는다.
- 핵심 씨앗은 하루 3개를 초과하지 않는다.
- 첫 사용자가 가입 없이 5분 안에 첫 완료까지 간다.
- 펫방에서 펫이 화면 중심에 크고 선명하게 보인다.
- 미완료, 복귀, 실패 상황에서 죄책감 문구가 없다.
- iOS/Android 실제 기기에서 주요 플로우가 깨지지 않는다.
- 개인정보 처리방침과 실제 수집 데이터가 일치한다.
- 첫 BM이 진화 성능이나 재시작 욕구를 과금화하지 않는다.

## 하지 않을 것

1차 MVP에서는 다음을 하지 않는다.

- 소셜, 친구, 랭킹
- 유료 랜덤 알 또는 가챠
- 진화 속도 2배권
- 리셋권 판매
- 펫 사망
- 강한 연속 기록 패널티
- 팀 협업 기능
- AI 자유 채팅
- 방 꾸미기 풀기능
- 여러 개의 유료 상품 동시 출시

## 첫 BM 원칙

첫 BM은 매출 극대화가 아니라 결제 의향 검증이다.

좋은 첫 BM:

- 비 오는 공부방 테마
- 별솜 새 여정 팩
- 작은 소품 또는 리본
- 프리미엄 진화 연출

나쁜 첫 BM:

- 진화 점수 판매
- 진화 속도 판매
- 새 여정 시작권 판매
- 실패 복구권 판매
- 랜덤 뽑기
- 놓치면 손해라는 압박 판매

## 최종 실행 순서

1. 문서 기준을 고정한다.
2. Expo 앱을 scaffold한다.
3. 도메인 타입, 점수, 진화, 생애, 대사 로직을 먼저 만든다.
4. SQLite schema, migration, repository transaction을 만든다.
5. 온보딩, 오늘, 펫방, 기록, 상점 placeholder를 연결한다.
6. QA 체크리스트로 첫 루프와 첫 진화를 반복 검증한다.
7. 솜몽 placeholder를 실제 사용 가능한 수준의 대표 아트로 교체한다.
8. EAS preview build를 만들어 실제 기기에 설치한다.
9. 무료 베타 사용자에게 배포한다.
10. 분석과 피드백으로 온보딩, 펫방, 진화 기대감을 고친다.
11. 첫 BM 상품 1개를 준비한다.
12. RevenueCat으로 구매, 복원, 권한 반영을 검증한다.
13. 첫 BM 결과로 계속 확장할지 판단한다.

## 마지막 기준 문장

TodoSom은 사용자를 더 완벽하게 만들기 위한 앱이 아니다. 사용자가 다시 시작할 수 있게 만드는 앱이다. 우리는 체크박스를 생명체로 번역한다.
